# Context

## Glossary

- `Owner`: the single operator who approves permissions and manages skills for this Felix instance.
- `Session`: a persisted interaction tied to one source thread and one Codex run history.
- `Session record`: the on-disk durable representation of a `Session`, stored under `workspace/records/sessions/<source>/`.
- `Thread key`: an opaque stable source-thread identifier produced by a source adapter; core modules store and index it but do not parse it.
- `Thread query`: read-only lookups over the session store. `resolvePendingPermissionThread` in `src/slices/approvals/resolve.ts` owns the precedence for locating the thread an `Owner decision` targets (by thread key, by owner-message anchor, or the un-anchored fallback); it lives in the approvals slice because it exists solely to serve approval resolution. Thread lookup by key is `findThreadHandle` in `src/slices/sessions/index.ts` (source-scoped index lookup) — neither the engine nor the owner console (`loadSessionDetail`) re-lists all handles to find one.
- `Session transitions`: the named, atomic mutations of session state in `src/slices/sessions/index.ts` — `shiftNextEvent`, `requeueEvent`, `recordTurn`, `clearCodexSession`, `setThreadBusy`. Each owns one load-mutate-save; the engine expresses intent through them rather than hand-writing load/mutate/save. The only direct `saveSessionState` left in the engine is the bulk queue filter in `sanitizeThreadQueue`.
- `Event schema`: the single home for the thread event-kind union in `src/slices/events/index.ts`. The `type` frontmatter discriminant (`source_event`, `felix_reply`, `owner_permission`, `permission_request`) is defined once. `buildEventFile` produces a kind's on-disk shape (frontmatter, body, transcript lines, file-name slug); `parseEventFile` reads one back as a typed `ParsedEvent`; `toUniversalEvent`, `eventAt`, and `historyTitle` are the readers' shared decoders. `thread-store` owns only the IO (`writeThreadEventAt`/`writeThreadEvent`); the engine and owner console no longer hand-decode frontmatter, and no writer re-encodes a kind inline.
- `Source intake`: the source-neutral workflow that receives a source adapter's normalized source packet, persists raw source evidence, routes an `Owner decision` when the packet is from the `Owner`, and otherwise hands the event to the engine.
- `Mention grammar`: how a Mattermost bot name becomes an `@mention` token, owned once by `src/adapters/mattermost/mentions.ts` (`mattermostMentionToken(s)`, `normalizeMattermostName`). The matcher (`isMentioned`) and source-owned turn context (`getTurnContext`) both build tokens from it inside the Mattermost adapter, so the instruction Felix is given can never drift from what the adapter actually matches. The Mattermost adapter re-exports the token builders for external callers.
- `Harness`: the execution backend that runs a session. The `Harness` port is defined in `src/core/ports.ts`; `CodexHarness` in `src/adapters/codex/` is the sole adapter. The port takes intent — `TurnInput {thread, event, eventFile, contact, skills, resumed}` — and returns a `TurnResult` with `parsed: ParsedAgentOutput` and `success: boolean`. The engine holds a `Harness` field injected at construction; it never imports the Codex adapter directly. Adding a second harness (Claude, local model) means a new file in `adapters/`, no core edits. `FakeHarness` in `tests/helpers/` makes the engine loop unit-testable without spawning a process.
- `Project`: a checked-out target repository Felix can change on behalf of the Owner. The workspace stores projects under `workspace/projects/<provider>/<namespace>/<repo>/`; provider is the host family (`github`, `gitlab`), namespace is the account or group, and repo is the repository name. Git adapters should treat this as the seam for branch, diff, PR/MR, and review work.
- `Turn decision`: the pure dispatch logic for a single harness attempt, in `src/core/decide-turn.ts`. `decideTurnResult(result, resumed, retriedFreshStart) → TurnDecision` converts a `TurnResult` into one of: `retry_fresh`, `fail`, `reply`, `no_skill`, `permission_required`, `fallback`. No IO, no logging — the engine shell executes each case through ports. Tested directly in `tests/decide-turn.test.ts` without a running engine or temp dirs.
- `Event routing`: the pure accept/filter predicates in `src/core/routing.ts`. `shouldAcceptEvent(event)` — DMs always accepted, channel posts require `mentions_bot`. `isOwnMattermostMessage(event, botUserId)` — filters the bot's own posts from the processing queue. Both are thin-wrapped by the engine's private methods; tested directly in `tests/routing.test.ts`.
- `Skill`: an installed capability defined by a `SKILL.md` file on disk.
- `Skill ID`: the directory name under `workspace/catalog/skills`; frontmatter `id` mirrors it for readability.
- `Agent runtime image`: the container runtime Felix uses to execute harness turns and skills, intentionally carrying common language and utility tooling for predictable session work while leaving provider-specific operational CLIs to explicit installation or skill setup.
- `Runtime capability`: a stable category of work the Agent runtime image guarantees for skills, such as Node execution, Python execution, data analysis, chart generation, document/PDF utility work, shell/network utilities, or Git/project editing.
- `Runtime package`: a concrete installed package or executable that currently implements a `Runtime capability`; package lists are operational details and may change without redefining the capability.
- `Shared runtime tooling`: owner- or skill-installed executables and user-space language packages stored under `workspace/runtime/` so they are available to later Felix sessions without mutating bundled skills, session records, or the base image filesystem.
- `Core data stack`: the provider-neutral Python data and visualization packages available in the agent runtime image for common reporting and artifact generation, with dependency expansion handled as shared runtime tooling rather than per-session virtual environments.
- `Document conversion suite`: heavy office-document conversion tooling such as LibreOffice; excluded from the initial Agent runtime image because the cost is large relative to the current guaranteed runtime capabilities.
- `Browser automation runtime`: browser binaries and their system dependencies for Playwright or similar browser automation; excluded from the initial Agent runtime image until bundled skills require in-container browser execution.
- `Source API posting`: direct agent use of the current source platform API to post intermediate messages or generated session artifacts in the active thread before the final reply. Mattermost is the current concrete source; future sources such as Slack must provide their own source-specific posting instructions.
- `Monthly cost summary report`: an AWS Jala reporting use case that summarizes the latest completed month and compares it with the previous completed month.
- `Report artifact bundle`: the files produced by a reporting use case as its primary output, usually including source data, visualizations, and a written summary.
- `Session view`: inspect-only in the first version; no manual rerun control.
- `Skill editor`: can edit name, description, permissions, and body.
- `Landing page`: simple list of sessions grouped by thread.
- `Session detail`: shows session history and raw artifacts; raw logs are behind the detail view. The `/api/sessions/{key}` response is the single source of truth — it carries `artifacts` inline, so there is no separate `/artifacts` endpoint for the UI to call.
- `UI data access`: internal JSON endpoints served by the existing app.
- `Contact editor`: edits the existing per-source contact record only.
- `Approval registry`: the authoritative store of approval records (`src/slices/approvals/registry.ts`, public face `src/slices/approvals/index.ts`). The slice also owns the owner-decision grammar (`decision.ts`), pending-thread resolution (`resolve.ts`), and decision orchestration (`apply.ts`)., written by the engine at request- and decision-time. The single source of truth for approvals — the UI and owner console read from it; nothing re-derives approvals from event files.
- `Approval requests`: visible in the UI with history of requested, approved, and rejected permissions.
- `Approval action`: the Owner can approve or reject approval requests from the UI.
- `Permission grant`: the additive contact authorization an `always` decision produces — one skill plus a set of permissions, de-duplicated against what the contact already holds. `decideApproval` names the grant intent (`PermissionGrant` in `src/approvals.ts`); `grantPermissions` in `src/contacts.ts` is the single home for the merge-and-persist. Nothing else merges contact grants.
- `Owner decision`: a typed owner verdict on a pending request (`OwnerDecision` in `src/types.ts`) — a `mode` (`once`/`always`/`reject`), the decider, and a `target` locating the pending thread (by `threadKey` from the UI, or by source-neutral owner-message anchor from an adapter). Transport-independent: adapters build it; the engine replays it. The reply grammar lives in `src/slices/approvals/decision.ts`, not in any adapter. Applying a decision is `applyOwnerDecision` in `src/slices/approvals/apply.ts`: it resolves the target thread, records the verdict (`decideApproval`), and on an `always` decision persists the `Permission grant` (`grantPermissions`) — the one place that turns grant intent into a stored grant, so `approvals` stays decoupled from `contacts`. The engine only queues the decision event and runs the turn.
- `Owner decision routing`: the source-neutral intake step that turns an Owner's platform-specific reply or reaction into an `Owner decision` target, only when it locates a matching pending approval.
- `Artifact viewer`: raw artifacts are shown in read-only markdown/text views.
- `Navigation`: top-level tabs for `Sessions`, `Skills`, `Contacts`, and `Approvals`.
- `Data fetching`: React `useQuery`-style client fetching with automatic refresh.
- `Auth`: owner login with an httpOnly session cookie.
- `Audit view`: simple history panel for owner edits and approval actions.
- `Exposure`: internal-only operator UI, not a public surface.
