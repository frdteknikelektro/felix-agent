# Felix 0.1.1 production release checklist

Publication is blocked until every item below is checked against the exact candidate digest. Store only sanitized evidence; never retain credentials, OAuth URLs, customer messages, phone numbers, prompts, or raw sensitive logs.

## Automated candidate gate

- [ ] Root and web `npm ci`, dependency-tree validation, and audits pass.
- [ ] All tests (at least the established 517-test baseline), typecheck, lint, and production build pass.
- [ ] Both Compose files, release-contract tests, scan-policy tests, and actionlint pass.
- [ ] AMD64 and ARM64 candidates build under one immutable registry digest.
- [ ] The exact AMD64 and ARM64 candidate images pass automated health, login, unauthenticated API/SSE rejection, disabled-source, read-only rootfs, restart persistence, graceful shutdown, and crash-restart smoke tests.
- [ ] Full Trivy JSON, VEX-filtered JSON, SARIF, secret scan, misconfiguration report, SPDX SBOM, CycloneDX SBOM, and GitHub attestations exist for that digest.
- [ ] Risk policy has zero blockers and zero VEX validation errors; suppressed findings remain in policy evidence.

## Runtime and lifecycle gate

- [ ] On both architectures: health, login, unauthenticated API/SSE rejection, disabled sources, read-only root filesystem, restart persistence, graceful shutdown, and crash restart pass.
- [ ] Clean container setup creates `/config/.env` with restrictive permissions.
- [ ] A backed-up `0.1.0` Workspace restores on a fresh host, upgrades to this digest with all state intact, and rolls back after a simulated failed upgrade.

## Source and harness matrix

- [ ] Mattermost: DM, mention, reply, reaction, attachment, approval, unauthorized rejection, and duplicate rejection.
- [ ] Discord: DM, mention, reply, reaction, attachment, approval, unauthorized rejection, and duplicate rejection.
- [ ] Slack: DM, mention, reply, reaction, attachment, approval, unauthorized rejection, and duplicate rejection.
- [ ] WhatsApp: DM, group mention, reply, reaction, attachment, approval, unauthorized rejection, duplicate rejection, and authenticated webhook behavior while disabled/reconnecting.
- [ ] Telegram: DM, group mention, reply, reaction, attachment, approval, unauthorized rejection, duplicate rejection, polling, and authenticated webhook mode.
- [ ] Codex, OpenCode, and Claude Code each pass normal completion, permission/retry, cancellation, provider failure, and recovery.

## Skills, operations, and publication

- [ ] Every dynamically discovered bundled skill passes metadata, loading, prerequisite, smoke, and permission-boundary checks; Google Workspace uses controlled credentials.
- [ ] Retention, redacted log collection, support/no-SLA, webhook exposure, VEX, backup timing, restore, upgrade, rollback, and `0.1.0` supersession docs are reviewed.
- [ ] A sanitized completed copy of `docs/releases/0.1.1-evidence.md` contains no unchecked item or placeholder, identifies the exact run ID/commit/digest, and is supplied through the publish workflow's `manual_evidence` input without amending the candidate.
- [ ] Manual publish verifies candidate run ID, digest, commit, version, and complete evidence before creating `v0.1.1`, Docker `0.1.1`, and the GitHub Release.
- [ ] Manual publish recomputes scan policy, verifies index provenance and both platform SBOM attestations, compares attested predicates with the attached SPDX files, and records every pre-manifest evidence asset hash.
- [ ] Any resumed source tag, image tag, GitHub Release, or existing release asset exactly matches the accepted commit, digest, metadata, and bytes.
- [ ] Published Docker tag, GitHub Release, scans, SBOMs, provenance, and acceptance all identify the unchanged candidate digest.
- [ ] Separate manual promotion verifies the Docker `0.1.1` digest before creating `latest`.
