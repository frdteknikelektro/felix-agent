# Felix 0.1.0 customer-ready release checklist

This checklist is the release gate for the first stable Felix image.

## Release identity

- [x] Set the canonical version to `0.1.0` in `package.json` and any customer-visible metadata.
- [ ] Create Git tag `v0.1.0`.
- [ ] Publish `frdinawan/felix-agent:0.1.0`.
- [ ] Publish or update `latest` only after all release checks pass.
- [ ] Record the immutable image digest in the GitHub Release.

## Support promise

The `0.1.0` support matrix is:

- Deployment: single-host Docker Compose.
- Architectures: `linux/amd64` and `linux/arm64`.
- Sources: Mattermost, Discord, Slack, WhatsApp, and Telegram.
- Harnesses: Codex, OpenCode, and Claude Code.
- Bundled skills: every Skill currently shipped under `skills/`, including `9router`, `database`, `felix-browser`, `general`, `install-tool`, `listen-speak`, `memory`, `office-documents`, `skill-creator`, `software-development`, `ssh`, `task-manager`, `template-skill`, and `usage-report`.

Database engines are capabilities of the bundled `database` Skill. They are not Felix's internal persistence layer. Felix stores its own Workspace data on the persistent filesystem volume.

## Build and dependency gate

- [x] `npm ci` completes for the root project.
- [x] `npm --prefix web ci` completes.
- [x] `npm test` passes locally.
- [x] `npm run lint` passes locally.
- [x] `npm run build` passes locally.
- [x] Production dependency audit has no high or critical findings.
- [x] Full root and web dependency audits report zero vulnerabilities.
- [x] No moderate findings remain; future findings require a written Security exception, mitigation, owner, and review date.
- [ ] Docker image builds with target `runtime`.
- [ ] Both `linux/amd64` and `linux/arm64` images build and start.
- [ ] Image scan, SBOM, provenance, and digest are attached to the release.
- [x] Downloaded third-party binaries have pinned SHA-256 checksums and base-image dependencies have a documented update policy.

The dependency findings identified during release preparation were remediated with patched transitive versions and updated development tooling. Root and web audits now report zero vulnerabilities.

## Security and data handling

- [ ] Require a strong, unique `OWNER_UI_SECRET`.
- [ ] Keep credentials outside the image and outside committed files.
- [ ] Confirm `DB_ENCRYPTION_KEY` backup and rotation guidance.
- [ ] Verify owner-console API and SSE routes reject unauthenticated access.
- [ ] Bind the service privately by default; do not expose port `53318` directly to the public internet.
- [x] Document customer-managed HTTPS for remote console access.
- [x] Enable `OWNER_UI_SECURE_COOKIE` for HTTPS reverse-proxy deployments.
- [ ] Ensure public Telegram and WhatsApp webhooks use webhook authentication secrets.
- [x] Add request-body limits and login rate limiting before public exposure.
- [ ] Review logs for tokens, prompts, credentials, and sensitive customer content.
- [ ] Document that Felix collects no Felix-owned telemetry.
- [ ] Document external data flows to configured messaging and model providers.
- [x] Add `LICENSE`, `SECURITY.md`, and third-party notices.

## Core runtime acceptance

Run these tests from a clean host using only the published image and [docker-compose.image.yml](../../docker-compose.image.yml):

- [ ] Run the setup wizard successfully.
- [ ] Start the service with a persistent Workspace volume.
- [ ] Confirm `/healthz` returns `{"ok":true}`.
- [ ] Log into the owner console.
- [ ] Confirm unauthenticated API and SSE requests are rejected.
- [ ] Confirm all disabled sources remain inactive when credentials are absent.
- [ ] Confirm restart preserves sessions, approvals, audit history, skills, credentials, and runtime state.
- [ ] Confirm graceful shutdown drains active work.
- [ ] Confirm a failed container restarts cleanly.

## Source acceptance

For each source, record a passing test with a real test account or controlled fixture:

- [ ] Mattermost: direct message, mentioned channel message, reply, reaction, attachment, and owner approval.
- [ ] Discord: direct message, mentioned channel message, reply, reaction, attachment, and owner approval.
- [ ] Slack: direct message, mentioned channel message, reply, reaction, attachment, and owner approval.
- [ ] WhatsApp: direct message, group mention, shared-number behavior, attachment, webhook authentication, and owner approval.
- [ ] Telegram: direct message, group mention, attachment, polling, webhook authentication, and owner approval.

For every source, also verify unauthorized contacts cannot invoke skills and that duplicate platform deliveries are not processed twice.

## Harness acceptance

- [ ] Codex harness completes a normal turn, permission request, retry, cancellation, and failure recovery.
- [ ] OpenCode harness completes the same scenarios.
- [ ] Claude Code harness completes the same scenarios.
- [ ] Model-provider credentials and model selection are documented for each harness.
- [ ] Provider failures produce a useful owner-visible error and do not corrupt Workspace data.

## Bundled skill acceptance

For every Skill under `skills/`:

- [ ] Metadata and permissions validate.
- [ ] The Skill loads after a clean startup.
- [ ] Its local smoke test passes.
- [ ] Its prerequisites, credentials, and expected external services are documented.
- [ ] Its permission boundary is tested.

When customer-owned prerequisites are available, run the corresponding prerequisite integration test. Do not place customer credentials, browser sessions, or external service state in the image.

## Data lifecycle acceptance

- [ ] Back up the complete Workspace data volume.
- [ ] Restore the backup to a fresh host.
- [ ] Verify sessions, approvals, audit history, skills, credentials, and runtime state after restore.
- [ ] Upgrade from the previous supported image with the same Workspace data.
- [ ] Verify migrations complete and old data remains usable.
- [ ] Roll back to the previous image after a failed upgrade.
- [ ] Document the backup timing, restore command, rollback command, and `DB_ENCRYPTION_KEY` requirement.

## Customer documentation

- [ ] Installation guide uses the versioned image tag.
- [ ] Configuration guide covers all supported sources and harnesses.
- [ ] Skill prerequisites are listed.
- [ ] Network exposure and HTTPS guidance is explicit.
- [ ] Data handling, retention, and external provider flows are documented.
- [ ] Backup, restore, upgrade, and rollback procedures are documented.
- [ ] Troubleshooting and log collection guidance is documented.
- [ ] Best-effort support channel is documented; no SLA is implied.
- [x] Apache-2.0 license and third-party notices are published.
- [x] `CHANGELOG.md` is complete; GitHub Release notes remain pending publication.

## Publication sequence

1. Clear all release blockers, especially high/critical vulnerabilities.
2. Run the full release gate and save the Release evidence.
3. Build and publish the multi-architecture `0.1.0` image.
4. Verify the published image by digest on a clean host.
5. Create Git tag `v0.1.0` and publish GitHub Release notes.
6. Update `latest` only after digest verification succeeds.
7. Announce the release with upgrade, backup, and rollback instructions.

## Evidence to retain

- Test, lint, and build logs.
- Source, harness, skill, and lifecycle acceptance results.
- Image digest, SBOM, provenance, and scan report.
- Dependency audit and any Security exceptions.
- Clean-install and upgrade/rollback records.
- Final documentation review.
